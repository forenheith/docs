Preliminary Recommended Sequence (Based on Confirmed Facts)

Based on already confirmed facts in this story (code, DB, production telemetry, and linked Jira plans), the following sequence is recommended as a preliminary implementation plan.

  1. Critical Priority: Explicit decision from Victoria Lewis to eliminate clear-text password storage.
    Current state is objectively high-risk: clear-text password data remains populated at scale. This must be formally approved for removal by Victoria Lewis because, while the technical implementation is relatively straightforward, it will significantly change established support login workflows. Code evidence also shows login-as-user/impersonation-related paths that depend on plaintext password data (for example tb_password and cp_password usage in legacy login flows), so once plain-text passwords are removed these scenarios will not work in current form and require a dedicated redesign/re-architecture scope.
  2. Create and approve a bounded scope for EQU-3257 deliverables.
    Freeze concrete outputs (artifacts, owners, sign-off criteria, timeline) to prevent unlimited scope growth.
  3. Execute duplicate-account program using existing phased plan references.
    Use EQU-3061 implementation planning (including implementation-plan discussion in EQU-3061) as the baseline: complete cleanup window logic, user communications, and enforcement checkpoint.
  4. Create Jira tasks from the approved phased plan.
    If Victoria Lewis confirms the EQU-3061 phased plan, create and prioritize the corresponding Jira work items for implementation and tracking.
  5. Complete targeted RCA for session/payment disruption scenarios.
    Use incident-anchored cases only (time/provider/path) and produce fix recommendations with measurable acceptance criteria.
  6. Finalize passkey rollout as an additive auth method.
    Keep it explicitly framed as an additional sign-in method that improves UX/security; do not treat it as automatic replacement of existing password rules.
  7. Run controlled security-hardening migration after approval.
    Execute approved auth-transition and support-workflow updates in controlled phases with rollback checkpoints.
  8. Close with measurable post-release verification.
    Define and track target metrics: auth failure rate, password-reset failure rate, duplicate-account trend, token-refresh error trend, and payment flow stability trend.

Important: the critical password-storage decision should be formally approved first, otherwise downstream scope and timelines remain unstable.
